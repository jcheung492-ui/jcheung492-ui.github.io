# 微信小程序迁移计划

> 张百星 / Justin 个人主页 → 网页 + 微信小程序双端
> 目标：内容一处发布、网页和小程序两端生效
> 定稿日期：2026-06-27

---

## 一、已确定的方案

| 决策项 | 选择 |
|---|---|
| 小程序前端 | **网页保持不动，小程序用原生 WXML/WXSS 单独做，只共享内容数据** |
| 内容与媒体托管 | **微信云开发 CloudBase**（免 ICP 备案、免域名白名单） |
| 小程序账号主体 | **个人主体** |

---

## 二、现状架构（网页）

- **UI**：纯手写 HTML/CSS/原生 JS，`index.html` 直接挂 17 个全局脚本（player、lyrics、imgedit、gallery 等），大量依赖 DOM / Canvas / HTMLAudio。
- **内容数据**：`js/published.js` 是一份"数据文件"（`window.SITE_PUBLISHED / SITE_GALLERY_PUBLISHED / SITE_JOURNAL_PUBLISHED / SITE_TEXT` 等），由管理面板自动生成；`js/data.js` 是内置默认作品。
- **发布流程**：管理面板（`admin.js` + `publish.js`）在浏览器里编辑 → 用 GitHub Git Data API 把 `published.js` + 封面提交到仓库 → GitHub Pages 重建上线。
- **媒体**：音频/视频走 Cloudflare R2（上传 Worker `justin-upload`），公开 r2.dev 地址读取。
- **评论/留言**：Supabase。
- **域名**：sound-truth.com（Cloudflare 灰云 + GitHub Pages）。
- **关键有利点**：内容本来就是数据，没和 UI 死绑 —— 让"内容一处发布、两端生效"工作量可控。

---

## 三、"一步覆盖网页 + 小程序"能做到什么程度

把两件事分开看：

| | 网页 | 小程序 |
|---|---|---|
| **内容发布**（改作品/随笔/文案/音频） | 即时 | **即时，免审核** ✅ |
| **代码 / UI 发布**（改版式、加功能） | 即时（Pages） | 必须走**微信审核**（几小时 ~ 1 天）⚠️ |

- **内容**：可做到真正"一步两端"。两端运行时都从同一个 CloudBase 数据源拉取，管理面板"发布"只写这一处，两端同时更新、小程序无需审核。**这是核心设计。**
- **代码**：可以做到"一条命令"（`git push` 推网页 + `miniprogram-ci` 上传并提审小程序），但小程序上线前**必然要过微信审核**——这是平台规则，去不掉。能做到"一键触发"，但不能让它像 Pages 一样秒上线。

## 四、关键问题与解决办法

### 问题 1：小程序不能跑 HTML / CSS / DOM
现有 player / imgedit / lyrics / gallery 全是 DOM / Canvas，小程序里跑不了。
**采用解法**：网页保持原样；小程序用原生 WXML 单独做一套 UI，**只共享内容数据**。改动小、风险低，代价是 UI 维护两套。

### 问题 2：域名白名单 + ICP 备案（最大坑）
小程序里 `request` / 音频 / 图片的域名必须是 HTTPS + 已 ICP 备案 + 在小程序后台白名单内。现有 `sound-truth.com`、`r2.dev`、`api.github.com` 全都不满足。
**采用解法**：**微信云开发 CloudBase**。其云数据库/云存储/云函数在小程序里调用**免 ICP 备案、免白名单**，直接绕过域名地狱。内容存云数据库，音频封面放云存储。

### 问题 3：账号主体与类目（注册期第一道闸）
个人主体不能用 web-view（本方案用不到，无影响）；**但音乐 / 作品展示类目对个人主体可能受限或需资质**。
**处理**：注册期**优先验证个人主体能否选到合适类目**（作品展示 / 文化娱乐）。过不了就要换主体，所以放在最前面。

### 问题 4：媒体改造
小程序音频用 `wx.createInnerAudioContext`（非 HTML5 `<audio>`）；imgedit 的 Canvas 裁切要换小程序 API。两端实现必然有差异。
**处理**：小程序侧重写音频播放、歌词同步、裁图逻辑。

## 五、目标数据流

```
                  ┌─────────────────────────────┐
   管理面板"发布" → │  CloudBase 云数据库 (内容)      │ ← 一处写入
                  │  CloudBase 云存储 (音频/封面)    │
                  └───────┬─────────────────┬─────┘
                          │ 运行时读取        │ 运行时读取
                   ┌──────▼──────┐    ┌─────▼────────┐
                   │  网页 (原样)  │    │ 小程序 (原生)  │
                   │ sound-truth │    │  WXML/WXSS   │
                   └─────────────┘    └──────────────┘
```

一次"发布"只写 CloudBase 一处 → 两端运行时各自拉取 → **内容即时两端生效、小程序免审核**。

## 六、各端职责

- **网页**：UI 一行不改。只把内容加载从"静态 `published.js`"改成"运行时从 CloudBase 拉 JSON"，`data.js` 内置数据保留作兜底/离线。媒体 URL 换成云存储地址。
- **小程序**：原生 WXML/WXSS 从零搭一套 UI，读同一份 CloudBase 数据；音频用 `wx.createInnerAudioContext` 重写，裁图等用小程序 API。
- **管理面板**："发布"目标从 GitHub 提交改为写 CloudBase（网页代码部署仍走 git）。

## 七、分阶段计划

### 阶段 0 — 注册与验证（先做，有外部等待）
- [ ] 注册小程序拿 AppID、开通云开发环境。
- [ ] **优先确认个人主体能选到合适类目**（作品展示 / 文化娱乐）。过不了整件事要换主体，放最前。

### 阶段 1 — 内容解耦（网页侧，不破坏现有线上）
- [ ] 在 CloudBase 建库表（作品 / 光影 / 随笔 / 文案，对应现有 `SITE_*` 结构）。
- [ ] 把现有 `published.js` 数据一次性导入云数据库。
- [ ] 媒体（音频/封面）从 R2 镜像到云存储。
- [ ] 网页加一层运行时 fetch，先和现有静态数据并存验证，跑通后切换。

### 阶段 2 — 改造发布器
- [ ] 管理面板"发布"改写到 CloudBase。
- [ ] 验证"一次发布、网页即时生效、刷新可见"。

### 阶段 3 — 小程序原生开发
- [ ] 搭页面骨架，读同一份云数据。
- [ ] 重写音频播放、歌词同步、图廊、裁图。

### 阶段 4 — 统一发布
- [ ] 一条命令：内容写 CloudBase（两端即时）+ 网页 `git push` + 小程序 `miniprogram-ci` 上传提审。
- [ ] 注：小程序代码上线仍需微信审核，此命令负责"一键触发"，审核闸去不掉。

### 阶段 5 — 联调上线
- [ ] 类目 / 审核走通。
- [ ] 灰度核对两端内容一致。
